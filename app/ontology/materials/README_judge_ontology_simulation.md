# README — Judge 온톨로지 시뮬레이션 재료 패키지
### 5개사 협상 시뮬레이션이 Judge 개발에서 하는 역할 · 차원과 가설 중심 (v1.0)

> **수신**: AI 박사님 · Judge 개발 담당
> **한 줄 요약**: 이 패키지는 해외 BD 전문가의 **판단 구조(암묵지)**를, 5개 실제 기업을 소재로 한 협상 시뮬레이션을 통해 **① 각 판단 차원에 무엇이 들어가는지(차원)** 와 **② 그것을 검증 가능한 명제로 뽑은 것(가설)** 으로 형식화한 재료다. Judge 온톨로지·CoT 데이터셋 설계의 **설계도**이며, 학습용 벽돌 자체는 아니다.
> **먼저 읽을 것**: 이 README → 각 기업 파일(`*_judge_ontology_material.md`) → 기획서 7장.

---

## 0. 이 패키지가 답하려는 질문

**"B2B 매칭에서 '이 둘이 될까?'를 전문가는 어떻게 판단하는가 — 그 판단을 기계가 배울 수 있는 구조로 어떻게 적는가?"**

Judge는 이 판단을 하는 엔진의 심장이다(기획서 7장). 문제는 이 판단이 Jin 같은 전문가의 **머릿속에만 있는 암묵지**라는 것. 이걸 모델에게 가르치려면 두 가지가 먼저 필요하다:

1. **차원(dimension)** — 판단을 어떤 축들로 쪼개는가, 각 축에 무엇을 보는가.
2. **가설(hypothesis)** — 각 판단을 "이런 상황에선 이렇게 판단한다"는 **검증 가능한 명제**로 뽑아낼 수 있는가.

이 패키지는 이 둘을 5개 도메인에 걸쳐 실증한다. **차원과 가설이 이 패키지의 두 기둥이다.**

## 2. Judge란 무엇인가 (최소 설명)

Judge = **후보 쌍(나 + 상대) 하나를 받아, 점수가 아니라 "근거·부족·리스크가 달린 구조화 판단"을 내리는 함수.** 실체는 전문가 CoT로 파인튜닝된 모델(EXAONE + LoRA)이며, 출력이 곧 사고 과정이다.

- **점수 폐기**: "87점"이 아니라 "조건부 진행 — 단 X 미확보 시 결렬, Y는 리뷰로 검증".
- **두 렌즈**: 같은 모델이 3파라미터(관점·목적함수·사전정보)만 바꿔 판매자/구매자로 작동.
- **결정 4종**: `recommend` / `conditional` / `hold` / `terminate`.
- **핵심 베팅**: 사실을 외운 게 아니라 **판단하는 방법**을 배웠기에, 학습에 없던 도메인(out-of-domain)에서도 같은 판단 구조가 작동한다.

## 3. 첫 번째 기둥 — 차원(Dimension)

### 3.1 차원이란
판단을 한 덩어리로 내리지 않고 **서로 다른 축으로 쪼개, 각각 독립 판정 후 종합**한다. 이 축 하나하나가 차원이다. 차원을 나누면 두 가지가 공짜로 생긴다 — 차원 간 불일치가 자동으로 확인 리스크가 되고(JDG-03), 전문가와 **어느 차원에서** 갈렸는지 평가가 가능해진다(카테고리 일치도).

### 3.2 7차원 + 게이트
| enum 키 | 차원 | 묻는 것 | 렌즈 |
|---|---|---|---|
| `industry_fit` | 산업 적합성 | 문제 도메인이 겹치는가 (업종 라벨 ≠ 도메인) | 공통 |
| `purpose_alignment` | 협업목적 정합 | 상대가 이걸 **원하는가** (want) | 공통 |
| `resource_complementarity` | 자원 보완성 | 내 것이 상대 결핍을 **메우는가** (fit) | 공통 |
| `stage_compatibility` | 사업단계 호환 | 규모·예산·타이밍이 현실적인가 | 공통 |
| `demonstrability` | 실증 가능성 | 증명할 신호·레퍼런스가 있는가 | 공통 |
| `substitute_comparison` | 대체재 비교 | 기존 대안보다 나은가 (상대평가) | buy 전용 |
| `opportunity_cost` | 기회비용 | 수용 시 무엇을 포기하나 | buy 전용 |
| `authenticity_gate` | 진위 게이트 | (인바운드) 상대가 실재·진짜인가 | 인바운드 앞단 |

각 차원 = `fit / caution / unfit` + 근거 1문장. 리스크는 3분류(선결/수익성/기각).

### 3.3 이 시뮬레이션이 차원에 대해 밝혀낸 것 (핵심 수확)

**차원은 고정된 체크리스트가 아니라, 구매자 유형·도메인에 따라 판정 기준이 이동하는 살아있는 구조다.** 19개 케이스가 이것을 실증했다:

**(a) 협업목적 정합의 기준 이동 — "구매자의 구매자가 누구냐"**
같은 판매자·같은 자산인데 정합을 판정하는 질문이 구매자 유형마다 이동한다:
```
최종소비자(브랜드) → 중간 클라이언트(대행사·유통사) → 미적 판단자(갤러리) → 
조직 ESG(CSR) → 사회적 미션(비영리) → 학술 검증(연구소) → 정책 성과(ODA)
```
→ 온톨로지 함의: 정합은 "판매자가 무엇인가"가 아니라 **"구매자가 이걸로 무엇을 하려는가"**로 판정. 중간자·조직·공공 구매자에서 이동 큼.

**(b) Impact 스펙트럼 — 같은 자산이 1순위이거나 감점 요인**
```
역신호 ←──────────────────────────────────────→ 1순위·핵심
갤러리      브랜드    대행사     CSR팀    사회복지센터/ODA
(앞세우면   (보너스)  (차별화)   (진짜)    (미션 자체)
 거부)
```
→ `impact_auto_first_checked`는 "체크했나"가 아니라 **구매자를 이 스펙트럼에 위치시키는 판정**. 에이엔폴리(전제 vs 핵심)·라이브케어(비용 vs Impact)에서 **아트가 아닌 도메인에서도 독립 재현**.

**(c) 실증 가능성의 도메인 의존성 — "보편 차원 + 산업별 신호" 2층 구조**
| 도메인 | 실증 신호 |
|---|---|
| 리빙 브랜드 | 실물 프린트 퀄리티 |
| 광고대행사 | IP·상업 사용권 clear |
| 갤러리 | 동시대 미술 담론 내 위치 |
| CSR/ESG·ODA | 수혜자 수·개발 성과 지표 |
| 비영리/재활 | 재활·교육 성과 지표 |
| 소재/화학 | 물성 인증 데이터(열안정성·VOC) |
| 의료기기 | 규제·보험 데이터(FDA·HCPCS·PL) |
| 식품 원료 | Novel Food 규제 상태 |
| 연구소 | peer-reviewed 학술 검증 (실증을 '생산') |
→ 모델이 배울 것 = "실물 확인하라"(암기)가 아니라 **"실증은 그 도메인의 신뢰 신호로 검증하라"(구조)**. 이것이 out-of-domain 전이의 핵심.

**(d) 차원의 새 형태·새 축 후보 (스키마 §8 Open)**
- 대체재가 경쟁사가 아니라 **구매자 자신의 R&D**(make-vs-buy, 코봇) 또는 **판매자 자신의 기회비용**(독점 수락, 코봇 C2).
- **결정 구조** — 담당자 ≠ 결정권자(CSR·소재 R&D). 협상 상대를 '내부 챔피언'으로 무장.
- **정보교환 순서** — NDA 선행 vs 샘플 선행(머쉬앤), IP·논문 발표권(연구소).
- **서비스형 보완재** — 상품이 아니라 역량·커리큘럼(비영리 PoC).

## 4. 두 번째 기둥 — 가설(Hypothesis)

### 4.1 가설이란, 왜 핵심인가
각 케이스의 판단을 **검증 가능한 명제**로 뽑은 것이 가설 카드다. 형식은 스키마 v1.1의 `hypothesis`:
```
recommendation_frame: exploit | explore    ← 확신 처방 / 검증할 베팅
lens: sell | buy
statement:       무엇을 주장/시험하는가
dimension:       어느 차원에 걸리는가
evidence_needed: 무엇으로 이 가설을 채점하는가
```

**왜 결론이 아니라 가설로 뽑는가 — 과적합 방지의 메커니즘이 여기 있다.**
- 판단을 "사실"로 적으면 모델은 결론을 **외운다**(다이브인 봇·키뮤 봇).
- 판단을 "검증 방법이 붙은 가설"로 적으면 모델은 **"판단이란 확인 리스크와 검증 계획을 남기는 행위"라는 구조**를 배운다.
- 이미 스키마에 그 자리가 있다: `risks[].check_method`, provenance의 `inferred/ask`, `decision: conditional`. 가설 카드는 이 필드들을 진짜 내용으로 채우는 일이다.

### 4.2 exploit / explore 가설의 구분 (기획서 7-B)
- **exploit(확신 처방)**: 이번 시뮬레이션에서 검증된 패턴 — "다음에 유사 상황이면 이렇게 하라". `hypothesis` 없이 프레임만.
- **explore(검증할 베팅)**: 아직 모르는 것 — "이 가설을 시험하려 이런 후보에 베팅한다". `hypothesis` 3요소 필수(무가설 explore는 스키마 위반, JDG-13).

### 4.3 가설의 3층 구조 (기획서 7-A.7, 검증 예정)
exploit/explore는 세 시간 스케일에서 작동한다 — 이 시뮬레이션은 세 번째 층의 관찰을 담고 있다:
| 층 | 스케일 | 채점자 |
|---|---|---|
| 후보 배분(7.4) | 배치 | 회신율(시장) |
| 추천 제시(7-B) | 추천 | 사람의 채택/이탈 |
| **협상 내 판단(7-A.7)** | **발화(라운드)** | **상대의 즉시 반응** |
협상의 매 수가 확신 손잡이/가설 손잡이로 나뉘고, 상대 반응이 가설을 채점해 확신으로 승격/폐기한다(케이스에서 관찰됨, 5개사로 검증 중).

### 4.4 이 시뮬레이션의 가설 → Judge 학습으로 가는 경로
```
시뮬레이션에서 추출한 가설 카드(explore)  
   → CoT 학습 데이터의 씨앗  
   → Judge가 유사 상황에서 그 가설을 '주입된 explore 라벨'로 재생산  
   → 사람의 채택·이탈이 그 가설을 채점(Strategy Loop)
```
**지금 만드는 가설 카드는 미래 Judge가 스스로 붙일 explore 라벨의 씨앗이다.** 이것이 차원(정적 구조)과 가설(동적 학습)이 만나는 지점이다.

## 5. 이 시뮬레이션이 Judge 개발에서 하는 역할

전체 흐름에서의 위치:
```
전문가 암묵지 (Jin의 BD 판단)
   │
   ▼  [이 시뮬레이션] ← 지금 여기
   │  차원에 무엇이 들어가는지 밝힘 + 판단을 가설로 형식화
   ▼
Judge 온톨로지 설계 (차원 정의·루브릭·산업별 신호·enum)
   │
   ▼
CoT 데이터셋 (작성가이드 §5 추론 무브 + 스키마 v1.1)  ← 실데이터 벽돌 투입
   │
   ▼
EXAONE + LoRA SFT  →  out-of-domain held-out 평가
```

**구체적 역할 셋:**
1. **온톨로지 스키마 설계 입력** — 각 차원의 판정 루브릭·산업별 신호 목록(§3.3)을 그대로 온톨로지 필드·enum으로.
2. **CoT 형판** — 리스크 3분류·결정 4종·거절 유형·진위 게이트가 실제로 어떻게 발현되는지의 예시.
3. **가설 라이브러리 시드** — explore 가설 카드가 Strategy Loop·추천 프레임의 씨앗(§4.4).

**이 시뮬레이션이 이미 밝혀낸 것 (과적합 방지의 직접 증거):**
같은 판단 구조가 **5개 서로 다른 도메인에서 독립적으로 성립**했다 — 정합 이동, Impact 스펙트럼, 전략적 역전, 카피 대응 분기, 채널 긴장. **구조가 도메인(표면)에 갇히지 않았다는 것**이 out-of-domain 일반화의 근거이자, 이 접근이 "특수 사례 → 보편 판단 구조"로 갈 수 있다는 증거다.

## 6. ⚠️ 주의사항 — 반드시 지킬 것

**이 재료를 그대로 학습(SFT)에 넣으면 안 된다.** 세 가지 이유:

1. **판매자 편중** — 도메인별로 판매자가 하나뿐(키뮤·에이엔폴리 등). 이대로면 "키뮤 봇"·"에이엔폴리 봇"을 학습(작성가이드 §0·§8의 과적합).
2. **구매측 전부 `[구성]`** — 상대 기업·private state는 AI(Claude)가 생성한 개연적 시뮬레이션. 실제 BD 관찰이 아니다. 자기 생성 데이터 학습 = **에코 챔버**(기획서 10.6).
3. **정답 앵커 없음** — 결말이 전부 시뮬레이션 산출. 실제 성사/거절 outcome(객관 체계, 평가검증_설계 §1)이 0.

**그러므로:**
- ✅ **써도 되는 것**: 온톨로지 차원 정의·루브릭·산업별 신호 목록 설계 / 가설 카드 형식 / 리스크·결정·거절 발현 형판.
- ❌ **쓰면 안 되는 것**: 이 대화·판정을 그대로 학습 코퍼스로 투입.
- 모든 구매측 항목은 provenance `[구성]` — 학습 시 신뢰 가중 하향 또는 실데이터 교체 대상.

## 7. 박사님 워크플로우 — 이걸로 무엇을 하면 되나

1. **차원 온톨로지 확정**: §3.3의 루브릭·신호 목록으로 각 차원의 판정 기준·산업별 하위 신호를 스키마로 못박는다. 새 축 후보(§3.3(d))의 편입 여부 결정.
2. **가설 라이브러리 구축**: 각 파일의 가설 카드를 스키마 v1.1 `hypothesis` 구조로 집계 → explore 가설 시드 라이브러리.

   - **A그룹 미팅 녹취 6건 증류** → `[관찰]` 등급 진짜 벽돌, outcome 앵커 후보.
