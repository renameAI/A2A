# 학습 스코어러(E9) vs new-judge 시스템 — 성능 비교

> 두 시스템을 **같은 9쌍·같은 풀**에서 실측 대조한 기록. 결론부터: **경쟁 관계가
> 아니라 순차 단계**이며, 비교해야 할 축은 "정확도"가 아니라 **역할과 단위비용**이다.
> 측정일 2026-07 · 스코어러 = E9(1.2B, ρ=0.789) · judge = gemini-3.1 온톨로지 협상 시뮬

---

## 1. 무엇을 비교하는가 (프레이밍)

| | 학습 스코어러 (E9) | new-judge |
|---|---|---|
| 답하는 질문 | "이 둘이 **얼마나 보완적인가**" (0–10) | "이 거래를 **진행해야 하는가**" (결정+근거) |
| 입력 | 공개 프로필만 | 공개 + **속내/봉인**(비공개 의도·제약) |
| 출력 | 연속 점수 (랭킹용) | recommend/conditional/hold/terminate + 근거·메일 |
| 파이프라인 위치 | `retrieve` 랭킹 **프리필터** | 최종 **판단** |
| 단위비용 | **36 ms · $0** | **2~10분 · $0.2~0.4** |

`training/README.md`가 처음부터 명시한 분업이다 — 스코어러는 Judge를 대체하지 않고
그 앞단에서 후보를 싸게 줄인다. 따라서 "누가 더 정확한가"는 잘못된 질문이고,
**"스코어러가 judge에게 넘길 후보를 제대로 추리는가"** 가 옳은 질문이다.

## 2. 직접 대조 — judge가 판정한 9쌍을 E9로 채점

judge 세션 9건에서 (판매사, 구매사, 결정)을 추출하고, **공개 프로필만** E9에 입력
(스코어러가 실전에서 보는 정보와 동일하게).

| judge 결정 | E9 점수 | 쌍 |
|---|---|---|
| conditional | 6.12 | 웨이비스 × 넥스틴 ← **유일한 in-distribution 쌍** |
| conditional | 5.46 | 하인텔 × 페이몬드 |
| terminate_structural | 4.47 | 키뮤스튜디오 × 트렌디코 어패럴 |
| conditional | 4.28 | 하인텔 × 우리기술 |
| recommend | 4.28 | 키뮤스튜디오 × 노르디스카 홈 |
| conditional | 3.83 | 코봇시스템 × 알펜모빌 AG |
| conditional | 3.76 | 코봇시스템 × 메디무브 GmbH |
| terminate_structural | 3.65 | 코봇시스템 × 유로마트 리테일 그룹 |
| hold | 3.50 | 키뮤스튜디오 × 갤러리 세이가 |

- **judge 결정서열 vs E9 점수: 스피어만 ρ = 0.467 (n=9)** — 약한 양의 상관.
- 진행(recommend/conditional) 평균 **4.62** vs 중단(terminate) 평균 **4.06** →
  분리 **+0.56** (방향은 맞으나 작다).

### 2.1 불일치는 결함이 아니라 설계다 (중요)

가장 큰 불일치인 **키뮤스튜디오 × 트렌디코 어패럴**(judge=terminate, E9=4.47로 높은 편)을
뜯어보면 두 시스템이 **서로 다른 질문에 각각 옳게** 답하고 있다:

- 공개 정보만 보면 "아트 스튜디오 × 아트 콜라보 캡슐을 내는 패션 브랜드" = **보완적**
  → E9가 높게 준 것은 **맞다**.
- 철수 사유는 buyer의 **봉인된 의도**("로열티 절대 불가, 시안만 받아 자체 재해석=카피").
  이건 공개 프로필에 없다 → judge만 볼 수 있다.

**코봇시스템 × 유로마트**(terminate_structural)도 같은 구조 — 철수 사유가 캐파 deal-breaker
라는 **숨은 구조 제약**이다. 즉 스코어러의 "높은 보완성"과 judge의 "진행 불가"는 **동시에
참**일 수 있다. 스코어러를 judge의 대체재로 쓰면 안 되는 이유가 실측으로 드러난 셈이다.

### 2.2 이 대조의 한계 (정직)

1. **n=9** — 통계적 결론 불가. 방향 참고용.
2. **입력 분포 불일치가 E9에 불리하다.** 프리셋 buyer의 공개 프로필은 1~2문장인데,
   E9는 ~2,700자 Gemini 리서치로 학습됐다(§EXPERIMENTS §4.5). 9쌍 중 학습 분포와
   맞는 것은 풀 기반 **웨이비스 × 넥스틴 1건뿐**이고, 그 쌍이 **최고점 6.12·argmax 7**로
   가장 확신 있게 나왔다 — 조건을 맞추면 신호가 뚜렷해진다는 방증.
3. judge 결정은 **정답이 아니다** — 상대 보고서 §5가 밝힌 대로 `outcome_anchor=false`
   (실제 성사/거절 앵커 0). 따라서 이 대조는 "정확도 측정"이 아니라 **두 관점의 일치도**다.

## 3. 규모·비용 — 결정적 차이 (실측)

E9로 judge 시스템의 풀(201곳)을 채점한 실측:

| 항목 | 실측값 |
|---|---|
| 처리량 | **28.0쌍/초** (쌍당 36 ms) |
| 200쌍 소요 | 7.1초 |
| **풀 전수 20,100쌍** | **11.9분**, 추가 비용 $0 (자체 GPU) |

같은 20,100쌍을 judge로 처리하면(상대 보고서 §2.7의 쌍당 2~10분·$0.2~0.4 기준):

| | 스코어러(E9) | judge |
|---|---|---|
| 20,100쌍 비용 | **$0** | 약 **$6,000** |
| 20,100쌍 시간 | **12분** | 약 **70일**(순차) / 17일(4워커) |

→ **스코어러가 있어야 judge가 풀 규모에서 실행 가능해진다.** 이것이 프리필터의 존재 이유다.

### 3.1 랭킹 품질 정성 확인 (무작위 200쌍)

| 상위 | 하위 |
|---|---|
| 6.37 서울반도체 × 지아이에스 | 2.58 미코 × 에스티큐브 |
| 6.33 모델솔루션 × 재영솔루텍 | 2.54 디젠스 × 예림당 |
| 6.33 미코 × 에프에스티 | 2.53 미코 × 코미팜 |

상위는 반도체·정밀제조 생태계 내 공급-수요 인접 쌍, 하위는 **반도체 × 출판**,
**반도체 × 동물약품**처럼 명백히 무관한 쌍 — 라벨 없이도 랭킹이 상식과 부합한다.

## 4. 리서치 파이프라인 비교 (둘 다 Gemini 그라운딩)

| | new-judge | 우리(E9) |
|---|---|---|
| 규모 | 200곳 | 1,000곳 |
| 모델 | gemini-3.1-flash-lite, max_tokens 2,000 | gemini-2.5-flash, max_output 4,000 |
| 소요 | 7분 (4워커) = **28.6곳/분** | ~58분 (8워커) = **13.8곳/분** |
| 비용 | $0.029 (200곳) | 미측정 |

**실행 가능한 개선**: judge 쪽 설정(flash-lite + 2,000토큰)이 워커당 처리량이 약 2배
높다. 우리 리서치도 같은 설정으로 낮추면 시간·비용을 절반 이하로 줄일 수 있다.
다만 우리는 더 긴 리서치(~2,700자)가 E9 학습 입력이라, **길이를 줄이면 E9 성능에
영향**을 줄 수 있으므로 재학습 전에는 바꾸지 않는다.

## 5. 결론과 권고

1. **둘을 파이프라인으로 결합한다** (대체가 아니라 직렬):
   `풀 20,100쌍 → E9 랭킹 12분 → 상위 k쌍만 judge($0.3×k)`.
   k=50이면 judge 비용 $15·수 시간 — 전수 대비 **400배 절감**.
2. **judge 결정을 스코어러 라벨로 쓰는 건 신중히.** 저렴한 judge 증류가 가능해 보이지만,
   judge 자체가 `outcome_anchor=false`라 **에코 챔버가 한 겹 더 쌓인다**(양측 보고서가
   공통 지적한 한계). 실제 성사/거절 앵커가 쌓인 뒤에 하는 것이 옳다.
3. **양쪽 보고서의 결론이 같다**: 병목은 코드가 아니라 **실데이터 앵커**다. judge는
   "실발송→실회신", 스코어러는 "실제 성사 라벨"을 기다린다 — **같은 데이터가 둘 다 푼다.**
