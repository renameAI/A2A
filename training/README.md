# EXAONE 관련도 스코어러 (training/scorer)

두 기업의 리서치 결과(또는 represent 온톨로지)를 입력하면 **관련도 0–10점**을 내는
경량 파인튜닝 모델. EXAONE 4.0에 **특수 토큰 + 선택적 unfreeze + FFN LoRA**로 학습한다.

> 📄 **실험 결과·방법론 상세(논문용)**: [EXPERIMENTS.md](./EXPERIMENTS.md) — 실측 수치
> 표, ablation, 발견(findings), 의사결정 로그, 재현 절차.

## 이게 무엇을 대체하나 (정직한 프레이밍)

Judge를 대체하지 **않는다.** 기획서·`judge_cases` README가 "점수 폐기"를 명시한다 —
Judge의 산출은 숫자가 아니라 근거·리스크가 달린 구조화 판단이다. 이 스코어러는 그 앞단의
**랭킹 프리필터**다: 현재 `app/engine/retrieve.py`의 `_score()`가 bigram overlap 휴리스틱으로
후보를 거르는데, 그 자리를 학습된 스코어러로 바꿔 수천 쌍을 싸게 정렬하고 top-k만 Judge로
넘긴다. 대칭 점수(관련도)는 "1차 랭킹"엔 정당하고, 비대칭 보완성 판단은 그대로 Judge 몫이다.

## 실험 결과 (실측)

> 전체 매트릭스·의사결정 로그·재현 절차는 [EXPERIMENTS.md](./EXPERIMENTS.md). 아래는 핵심
> 결과 요약이다. 지표는 held-out(**회사 단위 분리**, 교차 쌍 폐기, 누수 0)에서 예측 점수와
> 실제 점수의 **스피어만 순위상관 ρ**. held-out은 합성 60쌍 / 실데이터 49쌍(E5·E6은
> **동일한 49쌍**이라 직접 비교 가능).

### 1) 학습 매트릭스 (E1–E6)

| # | 모델 | 데이터 | LoRA | epoch | 학습시간 | 스피어만 ρ | 판정 |
|---|---|---|---|---|---|---|---|
| E1 | 1.2B | 합성 | FFN | 1 | 388s | **0.238** | 미학습 |
| E2 | 1.2B | 합성 | FFN+Attn | 3 | 1,670s | **0.698** | 배웠음 |
| E3 | 32B | 합성 | FFN+Attn | 3 | 3,848s | **0.668** | 배웠음 |
| **E4** | **1.2B** | **합성** | **FFN** | **3** | **1,163s** | **0.690** | **배웠음** |
| E5 | 1.2B | 실데이터(facts) | FFN | 3 | 694s | **0.753**† | 부분 배움 |
| E6 | 32B | 실데이터(facts) | FFN | 3 | 1,344s | **0.729**† | 판정 불가† |

관찰:

- **E1→E4 (attention은 이 태스크에 불필요 — 디렉터 원 스펙 지지).** E1(FFN-only·1ep)의
  실패(ρ=0.238)는 attention 부재가 아니라 **epoch 부족**이었다. epoch을 3으로 통제하자
  FFN-only(E4, ρ=0.690)가 FFN+Attn(E2, ρ=0.698)과 **사실상 동일**(Δρ=0.008). "FFN에만
  LoRA, attention 제외"라는 원 스펙이 실측으로 지지된다 — 파라미터도 더 적고 사전학습
  attention을 보존한다.
- **E2 vs E3 (1.2B ≈ 32B).** 합성 태스크에서 1.2B(0.698)와 32B(0.668)가 동급 → 이 태스크가
  32B의 표현력을 다 쓸 만큼 어렵지 않다. **온디바이스(1.2B) 배포의 근거.**
- **† 좁은 라벨 주의.** 실데이터 held 49쌍의 실제 점수가 `{2,3,5}`뿐(8점 이상 0개, §아래
  "실값 분석"). 그래서 "고관련(≥8) vs 저관련(≤2) 분리"를 정의할 수 없어 "판정 불가"로
  표기한다 — 측정 실패가 아니라 라벨 분포가 좁다는 뜻. ρ는 유효하다.

### 2) 파인튜닝 vs 제로샷 (E7) — **핵심 결과**

파인튜닝의 이득을 분리하려고, **라벨 생성과 동일한 프롬프트·temperature(0.2)·동일 held 49쌍**
으로 학습 안 한 모델들을 제로샷 채점했다(`zero_shot_eval.py`).

| 모델 | 학습 | 스피어만 ρ (vs 32B 라벨) | 비고 |
|---|---|---|---|
| EXAONE-4.0 **1.2B** | ❌ 제로샷 | **−0.294** | **역상관** (동전 던지기보다 나쁨) |
| K-EXAONE-236B (Friendli API) | ❌ 제로샷 | **0.705** | 49콜, 파싱 실패 0 |
| EXAONE-4.0.1 **32B** | ❌ 제로샷 | **0.961** | 라벨 만든 장본인 — 자기일치도(상한) |
| **1.2B + 특수토큰 FT (E5)** | ✅ 파인튜닝 | **0.753** | **전체 1위 (제로샷 API보다 높음)** |
| 32B + 특수토큰 FT (E6) | ✅ 파인튜닝 | **0.729** | 1.2B와 동급 (Δ는 노이즈) |

**읽는 법 (라벨 순환 주의):** 라벨 자체가 32B(temp0.2)의 산출이다. 그래서 —

- **32B 제로샷 0.961은 "자기 재현"**(라벨 만든 모델이 같은 프롬프트로 자기 판단을 재현하는
  상한선)이지 절대 품질이 아니다.
- **1.2B 제로샷 −0.294가 이 실험의 핵심 수치.** 같은 모델·같은 프롬프트에서 파인튜닝 전엔
  역상관이고, 특수토큰 파인튜닝 후 **−0.294 → +0.753 (Δρ>1.0)**. 이 태스크 능력은 파인튜닝이
  '개선'한 게 아니라 사실상 **만들어냈다**.
- **파인튜닝 1.2B(0.753) > 236B API 제로샷(0.705).** 공정한 표현은 "1.2B가 236B보다 똑똑"이
  아니라 — 파인튜닝 모델은 32B 라벨 분포를 직접 배웠으니(증류) 그 라벨과의 합치에서 유리 —
  "**26분의 1 크기·API 비용 0으로 교사(32B) 자기일치(0.961)의 78%를 회수**"다.
- **"1.2B가 극악인가?" → 아니오.** 같은 데이터로 32B를 학습(E6=0.729)해도 1.2B(E5=0.753)
  대비 이득이 없다. 병목은 **모델 크기가 아니라 입력 정보 밀도**(facts-only의 섹터 수준
  정보)다. 개선 경로는 모델 확대가 아니라 리서치 품질(웹검색 리서치 또는 represent 온톨로지).

### 3) 실값 분석 — 왜 1.2B 제로샷이 역상관인가

held 49쌍을 섹터 관계로 나누면 −0.294의 메커니즘이 드러난다:

| 그룹 | n | 라벨 평균 | 1.2B ZS | API ZS | 32B ZS | **E5 (FT)** |
|---|---|---|---|---|---|---|
| 같은 섹터 | 32 | 2.12 | **3.56** ↑ | 2.97 | 2.19 | 2.18 |
| 다른 섹터 | 17 | 3.24 | **2.35** ↓ | 4.29 | 3.24 | 2.96 |

루브릭은 "동종 경쟁사는 낮다"인데 **1.2B 제로샷은 같은 섹터를 더 높게** 준다 — 보완성이
아니라 **유사도를 채점**한다(방향이 정반대). 게다가 1.2B의 reason 텍스트는 *"동일 섹터에
속하지만 보완 관계는 확인되지 않아"* 라고 **말은 맞게 하면서 점수는 4를 준다** — 형식(JSON)은
따르는데 판단(루브릭)은 못 따르는 소형 모델의 전형.

쌍별 실값(대표 6쌍):

| 쌍 | 섹터 | 라벨 | 1.2B ZS | API ZS | 32B ZS | **E5** | E6 |
|---|---|---|---|---|---|---|---|
| 한세엠케이 × 시프트업 | 소비재 × 기술서비스 | **5** | 2 | 4 | 5 | **3.12** | 3.07 |
| BNK금융지주 × 시프트업 | 금융 × 기술서비스 | **5** | 2 | 7 | 5 | **3.11** | 3.12 |
| 삼성물산 × 효성첨단소재 | 산업서비스 × 공정 | **5** | 2 | 6 | 5 | **3.01** | 3.03 |
| BNK금융지주 × 유진투자증권 | 금융 × 금융 | 3 | **4** | 2 | 3 | **2.10** | 2.05 |
| 그린리소스 × 동일금속 | 광물 × 광물 | 2 | **4** | 3 | 2 | **2.08** | 2.01 |
| 비츠로시스 × HVM | 제조 × 제조 | 2 | 2 | 2 | 2 | **2.04** | 2.02 |

- **파인튜닝이 뒤집은 순간이 실값에 보인다.** 셋에서 라벨이 가장 높은 true=5 세 쌍에 1.2B
  제로샷은 전부 **최저점(2)**을 줬는데, 같은 모델이 파인튜닝 후(E5)엔 그 세 쌍을 정확히
  **49쌍 중 1·2·3위**(3.12/3.11/3.01)로 올려놨다.
- **E5·E6은 거의 같은 순위 함수를 배웠다** — 쌍별 순위 상관 **0.875**, 절대값도 쌍마다
  ±0.05 안쪽(3.12↔3.07, 3.01↔3.03). 크기가 26배 달라도 같은 데이터에서 같은 걸 배운다.
- **왜 값이 2~3점대에 몰리나:** facts-only 입력이라 라벨 자체가 좁고(§위 †), 예측도 그
  범위 안에서 미세한 순위만 나눈다. 그래서 **절대 점수가 아니라 순위(재랭킹)로만** 소비한다.

### 4) 서빙 (라이브)

승자 모델은 **E5(1.2B)** — ρ 최고(0.753)이면서 32B의 26분의 1 크기. 현재 배치:

```
[GPU 서버] serve.py (FastAPI, 127.0.0.1:8500 — E5 상주, /score-batch)
    ↑ SSH 터널 (방화벽 22만 허용): ssh -f -N -L 8500:localhost:8500 -i <key> tta@<host>
[엔진(맥)] scorer_client.score_batch()  ← A2A_SCORER_URL=http://localhost:8500
    ↑
retrieve.py: τ 게이트(=0.12)는 휴리스틱 유지 → 통과 후보의 '순서'만 학습 점수로
             재랭킹(창 64건) → CandidateOut.learned_relatedness → UI 🧠 칩
```

- **게이트/랭킹 분리:** 스코어러는 랭킹 프리필터일 뿐. 캘리브레이션된 τ 게이트·Judge 판단은
  건드리지 않는다. 서버 부재·타임아웃·응답 불일치 시 **None 폴백 → 휴리스틱 순서 유지**
  (조용한 대체 없음, 로그 명시). 단위 테스트 8건(mock HTTP) + 전체 스위트 311 통과.
- **E2E 실측:** 엔진 `/v1/retrieve` HTTP → 터널 → E5 채점 → 응답 `learned_relatedness: 2.13`
  (리비 하노이 함롱) 확인.

## 데이터 형식

`RelatednessPair` JSONL 한 줄:
```json
{"a_id":"co-1","a_text":"A 리서치 결과…","b_id":"co-2","b_text":"B 리서치 결과…","score":7,"mode":"research","source":"claude-opus-4.8"}
```
`mode`: `research`(리서치 결과) | `ontology`(represent 온톨로지). 같은 파이프라인으로 둘 다 학습.

## 실행 순서

```bash
# 0) (로컬) GPU 없이 데이터·설정·규모 검증 — 불균형·누수·스텝수 확인
python -m training.scorer.dry_run --pairs data/pairs.jsonl

# 1) (서버) 의존성 — 서빙 레포와 분리
pip install -r training/requirements-train.txt

# 2) (서버) 재사용할 미학습 토큰 17개 고르기 (아래 '서버 확인사항' 참조)
#    → {"score":[11개 문자열],"struct":[6개 문자열]} 를 tokens.json 으로 저장

# 3) (서버) 학습
python -m training.scorer.train \
  --pairs data/pairs.jsonl \
  --model-id /path/to/EXAONE-4.0-32B \
  --tokens-json tokens.json \
  --output-dir training/runs/scorer

# 4) (서버) 추론 — 기댓값 readout으로 연속 점수
python -c "from training.scorer.infer import RelatednessScorer; \
  s=RelatednessScorer('/path/to/EXAONE-4.0-32B','training/runs/scorer'); \
  print(s.score('A 리서치…','B 리서치…'))"
```

## 설계 근거 (전문가 리뷰 반영)

- **untied unfreeze** — EXAONE 4.0-32B는 `tie_word_embeddings=False`. 점수 토큰은
  *예측*되므로 lm_head(출력) 행이, 구조 토큰은 *읽히므로* 입력 행이 학습돼야 한다.
  안전하게 17개 특수 토큰의 입력·출력 행을 모두 unfreeze한다(1.2B은 tied라 자동으로 한 행렬).
  `requires_grad`는 텐서 단위라 행 일부만 못 푸므로, **backward 훅으로 특수 토큰 외 행의
  grad를 0으로** 눌러 17행만 갱신한다.
- **기댓값 readout** — 11개 점수 토큰을 독립 분류로 두면 CE가 "6 대신 7"과 "0 대신 7"을
  똑같이 벌준다(순서성 소실). 추론 때 점수 토큰 로짓만 softmax → 기댓값 Σp·k로 연속·보정 점수.
- **계층 샘플링 + 회사 분할** — 랜덤이면 99%가 0–2점이라 모델이 "무조건 낮게" 찍고도
  정확해 보인다. 점수 버킷 상한으로 균형을 맞춘다. held-out은 **회사 단위 분리**(교차 쌍 폐기,
  누수 0)라 회사명 암기가 아닌 판단 구조 전이를 측정한다.
- **FFN LoRA(기본) + 어텐션 ablation** — "두 기업 비교"는 본질적으로 어텐션 연산이지만,
  사전학습 어텐션이 이미 in-context 비교를 하므로 FFN-LoRA로 읽어내는 베팅이 합리적이다.
  단 가정이므로 `--include-attention`으로 ablation을 돌려 실측하라.
- **미학습 토큰 평균 초기화** — 노이즈 대신 기존 임베딩 평균에서 출발(수렴 가속).

## 서버 확인사항 (torch 파트는 로컬에서 검증 불가 — 반드시 서버에서)

1. **미학습 토큰 선택** — `model_setup.find_unused_token_candidates(tokenizer)`로 예약/미학습
   토큰 후보를 뽑아 사람이 확인 후 17개 채택. voca 102,400 중 실제 미사용 슬롯을 써야
   기존 능력을 안 건드린다.
2. **LoRA 타겟 이름** — 기본 `gate_proj/up_proj/down_proj`(SwiGLU 확인). EXAONE4 구현의
   실제 서브모듈명과 다르면 PEFT가 "타겟 없음"으로 명확히 에러 내니 `model.named_modules()`로
   확인 후 조정.
3. **분산 학습(ZeRO-3)** — 행 마스킹 grad 훅은 DDP엔 그대로 되지만, DeepSpeed ZeRO-3는
   파라미터를 분할하므로 훅 동작을 검증하고 필요시 조정. 대안: `LoraConfig(modules_to_save=
   ["embed_tokens","lm_head"])`로 임베딩 전체를 저장(단 17행만이 아니라 전체가 trainable).
4. **저장/복원** — 어댑터(LoRA) + `special_token_weights.pt`(학습된 17행)를 함께 저장하고
   `infer.py`가 둘을 복원한다.

## ⚠️ 정직성 경고

- **에코 챔버(실측 확인)** — 실제 라벨은 **로컬 EXAONE-4.0.1-32B**가 매겼다(무API 자기증류,
  당초 계획한 Claude/외부 API가 아님). 상한이 EXAONE 판단 품질이고 실제 성사 outcome 앵커가
  0이다(`judge_cases` README와 같은 위험). E7에서 32B 제로샷의 자기일치도가 **ρ=0.961**로
  측정됐는데, 이는 "라벨을 만든 모델이 자기 판단을 재현하는 상한"일 뿐 절대 품질이 아니다.
  절대점수보다 **순위**로 쓰고, 가능해지면 실제 성사 라벨로 교체한다.
- **좁은 라벨의 다이나믹 레인지** — facts-only(섹터 수준) 입력이라 실데이터 라벨이 `{2,3,5}`로
  좁다(§실험 결과). 강한 보완(8–10점)을 식별하는 능력은 이 데이터로 **검증되지 않았다** —
  그건 8점 이상 표본이 필요하고 이번 데이터엔 없다. 개선은 모델 확대가 아니라 리서치 품질.
- **외부 API 대량 호출** — 웹검색 기반 리서치·페어 스코어링을 외부 API로 돌린다면 큰 비용.
  데이터 생성 전 **AXR팀 협의** 필수. (이번 실데이터는 공개 CC0 데이터셋 + 로컬 EXAONE로
  완전 무API 처리했다.)
